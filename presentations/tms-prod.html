TMS as you know it

  • One WebApi application, one database.
  • Just Charm templates.
  • No email delivery stats.
  • One application performing multiple tasks (accepting, composing, sending).
  • One application - single point of failure.
  • Only specialized email types (e.g. Make, Change, etc).
  • Async processing dependent on .NET/IIS threads.

Actual TMS services

  • Six microservices (+ few supporting) and 3 databases.
  • Support for different template languages (Mustache, Handlebars).
  • Full email statistics (deliveries, bounces, opens, etc).
  • Small, specialized microservices.
  • We can survive downtimes.
  • Sending custom emails.
  • Full async processing with RabbitMQ.

TMS Iris

TMS Iris

  • In order to improve templates we needed to take ownership (and move away from Charm).
  • We couldn't take ownership without providing similar tool to edit templates.
  • Downsides of Charm:
    1. Limited editing abilities (no preview)
    2. Poor testing (different render engine for testing and for prod usage).
      1. Features of Iris:
        1. Easy template editor with preview.
        2. Built on top of TMS.
        3. Versions management.
        4. Allow to save changes without publishing to production.
        5. Easy publish and revert.
        6. (Planned) Extended testing.
        7. (Planned) Creating new templates with code completion.
        8. (Planned) Full render.
        9. Modular email support (integration with SRS).

TMS Iris - future

  • Code completion for tags
  • Full email rendering
  • Support for modular emails
  • Security

Event-based email sending

Event-based email sending

  • WHEN, WHAT and WHO?
  • This knowledge should be owned by TMS.
  • E.g. Reservation Service, OT WinService
Knowledge present in other services. OtWinService - 24h reminder, DFF and NoShow. If take ownership we need to understand how the process work and identify business owners (NoShow)

Event-based email sending - events

  • When reservation is created
    • Send confirmation email
    • Schedule 24h reminder (ForgetMeNot)
  • When reservation is seated
    • Schedule DFF email (ForgetMeNot)
  • When reservation is marked as NoShow
    • Send NoShow email

Rule Engine

Rule Engine in TMS

Rule Engine - example

						
rule "Schedule email reminder"
    on
        Reservation_Event_v1
    when
        [Event.Region] == 'na'
        [Event.ReservationState] == 1
        not [Reservation.IsRestRef]
    then
        scheduleEmail([Event.ReservationId], "24h reminder")
end
						
					
						
{
  region: "na",
  reservationState: 1,
  restaurantId: 123,
  confirmationNumber: 567
}

Rule Engine - new email type?

						
rule "Send welcome admin email and text"
    on
        User_Event_v1
    when
        [Event.Type] == 'UserRegistered'
        [User.IsAdmin]
    then
        sendEmail([User.Email], "USER-AdminWelcome")
        sendText([User.PhoneNumber], "SMS-AdminWelcome")
end
